# 互联网大厂缓存与数据库一致性方案对比

---

## 原理篇：缓存一致性的本质问题

### 1.1 为什么缓存和数据库会不一致？

缓存（Redis）与数据库（MySQL）是两个独立存储，无法原子更新。不一致的根源有三：

1. **双写非原子**：更新DB和更新/删除缓存无法在同一个事务中完成，中间必有时间窗口
2. **主从同步延迟**：MySQL主从复制存在延迟（通常ms级，高负载可达秒级），读从库可能读到旧数据
3. **并发竞争**：高并发下读写交错，可能出现：写请求更新DB后删缓存 → 并发读请求缓存Miss → 读从库旧数据 → 回填旧缓存（脏数据复活）

**典型脏数据复活时序**：

```
时间线  写请求(W)              读请求(R1)             读请求(R2)
T1     删除缓存key
T2                            缓存Miss，查从库
T3     更新主库data=v2
T4                            从库尚未同步，读到v1
T5                            回填缓存key=v1 ❌脏数据!
T6                            缓存命中，返回v1 ❌
T7     从库同步完成(延迟窗口)
```

> 核心矛盾：**写操作的"删缓存→更新DB"和读操作的"缓存Miss→查DB→回填缓存"是并发进行的，没有任何天然机制保证顺序**。

### 1.2 延迟双删详解

**延迟双删（Delayed Double Delete）** 是早期解决上述问题的经典方案：

```
流程：先删缓存 → 更新数据库 → 等待固定延迟（如500ms）→ 再删一次缓存
```

核心目的：第二次删除是为了清理延迟窗口内被并发读请求回填的脏缓存。

**时序示意**：

```
T0  删除缓存key（第一次）
T1  更新主库data=v2
T2  [等待500ms延迟窗口]
    -- 窗口内：并发读可能从旧从库读到v1，回填缓存
T3  删除缓存key（第二次，清除窗口内的脏回填）
T4  后续读请求缓存Miss，从已同步的从库读到v2，正确回填
```

### 1.3 延迟双删的4大致命缺陷

#### 缺陷1：延迟时间是"拍脑袋"，不可控

固定延迟无法适配主从同步、网络、负载的动态波动：
- 太短（如200ms）：主从延迟高峰期仍可能读到旧数据，第二次删除后又有并发读回填脏数据
- 太长（如2s）：缓存空窗期过长，大量请求穿透DB，加剧缓存击穿风险
- 生产环境主从延迟可能是10ms也可能是5s，**没有一个固定值能覆盖所有情况**

#### 缺陷2：一致性不可靠，脏读无法根除

- 延迟窗口内仍可能读到旧从库数据、回填脏缓存
- 第二次删除失败（进程宕机、消息丢失、网络异常）会永久残留脏数据，缺乏补偿机制
- 即使第二次删除成功，延迟窗口内的脏读已经对业务造成了影响

#### 缺陷3：高并发下性能灾难

频繁删缓存 → 大量缓存Miss → 请求穿透到数据库：
- 双删意味着每次写操作产生两次缓存空窗，读请求集中穿透DB
- 高并发写场景下，缓存几乎始终处于空窗状态，等同于没有缓存
- 极易引发**缓存击穿**（热点Key失效后大量并发穿透）甚至**缓存雪崩**

#### 缺陷4：业务侵入强、维护成本高

- 业务代码要嵌入延迟逻辑（Thread.sleep? 定时任务? MQ延迟消息?），各种实现都有坑
- 需要重试补偿机制：第二次删除失败怎么办？又引入新的复杂度
- 故障排查难：延迟窗口内的不一致几乎不可追踪，可观测性差

> **一句话总结：延迟双删是用静态时间窗口对抗动态系统不确定性，本质是权宜之计，不适合高并发、大规模分布式场景。**

## 方案篇：4种主流替代方案详解

### 方案1：Binlog 异步事件驱动

> 阿里/美团/字节主流方案 | 弱一致、高吞吐、0业务侵入

**核心思路**：放弃业务线程里的延迟删除，改为监听数据库变更日志（Binlog），**事件驱动、异步兜底**，彻底解耦业务与缓存维护。

**完整流程**：

```
业务线程                    Canal/MaxWell               MQ消费者
  |                              |                          |
  |-- 更新主库data=v2  -------->|                          |
  |   (生成Binlog事件)           |                          |
  |                              |-- 捕获Binlog ----------->|
  |                              |   投递到MQ               |
  |                              |                          |-- 消费消息
  |                              |                          |   异步删除缓存key
  |                              |                          |   (重试+死信队列保障)
```

**优势**：
- **0业务侵入**：业务代码无需感知缓存维护逻辑
- **高可靠**：Binlog是数据库原生机制，不会丢失；MQ消费失败有重试+死信队列兜底
- **可追溯**：每条Binlog事件有完整元数据（位点、时间戳、变更前/后值），便于审计和排查
- **无固定延迟**：删除时机由事件触发，而非拍脑袋的时间窗口
- **不阻塞主流程**：缓存维护完全异步，不影响写操作响应时间

**常见坑点**：
- **Binlog消费顺序**：同一Key的多次变更必须按顺序消费，否则可能出现旧变更覆盖新变更。解决方案：MQ分区按Key哈希，保证同Key有序
- **消费延迟**：MQ消费本身有延迟（通常秒级），在读请求到达时缓存可能尚未删除。业务需接受短暂不一致窗口
- **Canal高可用**：Canal实例宕机需要自动切换，建议部署Canal集群+HA模式

**失败恢复策略**：
- MQ消费失败 → 自动重试（3次）→ 进入死信队列 → 人工/定时任务补偿清理
- Canal宕机 → Canal集群HA切换 → Binlog位点自动恢复

**适用场景**：读多写少、允许短暂不一致（商品详情、用户信息、内容列表、配置数据）

### 方案2：版本号/时间戳校验

> 京东/拼多多常用 | 强一致兜底、防旧数据覆盖

**核心思路**：给数据加版本号（version）或更新时间戳（updated_at），**写缓存前先校验版本，只允许新版本覆盖旧版本，从根源阻止脏数据回填**。

**完整流程**：

```
写请求：
  更新DB: SET data=v2, version=2 WHERE id=X  (version+1)
  删除缓存key（无需延迟，只删一次）

读请求：
  缓存Miss → 查DB（读从库）
  比较版本：DB.version > 缓存.version？
    YES → 写回缓存key={data:v2, version:2}   ✅正确回填
    NO  → 丢弃旧数据，不写缓存                 ✅阻止脏回填
```

**时序验证**（主从延迟场景）：

```
T0  删除缓存key
T1  更新主库: data=v2, version=2
T2  读请求缓存Miss → 查从库（尚未同步）→ 读到: data=v1, version=1
T3  校验: DB.version(1) < 已知最新version(2)? → 丢弃，不写缓存 ✅
T4  从库同步完成，后续读请求读到: data=v2, version=2 → 写回缓存 ✅
```

**优势**：
- **彻底解决主从延迟导致的脏缓存**：旧数据因版本低被拒绝，永远无法回填
- **无需延迟等待**：只删一次缓存，无时间窗口
- **实现简单**：只需在数据表增加version字段 + 缓存存储中附带version

**常见坑点**：
- **版本号来源**：读请求如何知道"最新版本"？方案一：缓存存储旧版本号，读请求拿DB版本与缓存版本比较；方案二：主库更新后同步写入一个"最新版本标记"到Redis
- **缓存中version的存储**：缓存value需包含version信息，如 `redis.set(key, JSON.stringify({data, version}))`
- **写频率极高场景**：频繁删缓存仍有击穿风险，可结合逻辑失效标记方案

**失败恢复策略**：
- 缓存删除失败 → 版本校验机制兜底，旧数据不会被写入，下次读请求自然会重建正确缓存
- 写操作version+1失败 → DB事务回滚，不会产生不一致

**适用场景**：对一致性要求较高、写频率适中（订单、库存、账户余额）

### 方案3：租约（Lease）机制

> Meta/Facebook经典方案 | 防并发写击穿、轻量原子

**核心思路**：缓存Miss时，返回唯一租约Token，**只有持有有效租约的请求才能写缓存，其他请求等待或丢弃，避免并发回填脏数据**。

**完整流程**：

```
读请求：
  GET key → Miss → 返回Lease Token(唯一、有效期短如50ms)

  查DB → 带Token写缓存：
    SET key value IF token == current_lease_token  (原子操作)
    Token匹配 → 写入成功 ✅
    Token不匹配 → 写入被拒 ✅（说明已有新数据或租约已失效）

写请求：
  更新DB → 删除缓存key + 删除租约Token
  （阻断所有正在持有旧租约的并发写入）
```

**时序验证**（并发读场景）：

```
T0  读请求R1: 缓存Miss → 获得Lease Token=A
T1  读请求R2: 缓存Miss → 获得Lease Token=B (新租约，A自动失效)
T2  写请求W:  更新DB → 删除缓存 + 删除租约
T3  R1回填: 带Token=A写入 → Token已失效 → 写入被拒 ✅
T4  R2回填: 带Token=B写入 → Token已失效 → 写入被拒 ✅
T5  后续读请求: 缓存Miss → 获得新租约 → 查DB(v2) → 带新租约写入成功 ✅
```

**优势**：
- **轻量级**：不需要版本号字段，不需要额外存储
- **原子性**：Redis Lua脚本实现Token生成+校验，保证原子操作
- **防并发击穿**：只有持有有效租约的请求能写缓存，其他请求直接丢弃或等待，避免并发回填风暴
- **自淘汰机制**：新租约自动使旧租约失效，无需主动清理

**常见坑点**：
- **租约有效期设置**：太短（5ms）可能导致正常请求也无法写缓存；太长（500ms）延迟窗口过大。建议50-100ms，根据业务RT调整
- **租约被拒后的策略**：请求被拒后不能无限重试。策略一：直接返回DB数据（不缓存）；策略二：短暂等待后重试获取新租约
- **热点Key竞争**：极高并发下多个请求争抢租约，可能导致只有第一个请求成功缓存，后续请求被拒。这是设计预期行为，并非Bug

**失败恢复策略**：
- 租约Token丢失 → 自然过期，不会残留脏数据
- 写操作删除租约失败 → 租约自然过期（有效期短），不会长期阻断

**适用场景**：高并发读、热点Key场景（热门文章、热搜榜、限流计数器）

### 方案4：逻辑失效标记 + 异步补偿

> 美团/滴滴常用 | 高频写场景、写操作极快

**核心思路**：不用物理删除缓存，而是**打失效标记（is_deleted=1 / 版本标记）**，读请求发现标记直接回源DB、重建新缓存；后台异步批量清理失效缓存，避免频繁删缓存引发击穿。

**完整流程**：

```
写请求：
  更新DB → 更新缓存: SET key {is_deleted:1, old_data:v1}  (标记失效，不删除)

读请求：
  GET key → 发现is_deleted=1 → 回源DB → 写入新缓存 {data:v2, is_deleted:0}

异步清理：
  定时任务扫描标记is_deleted=1的缓存 → 批量物理删除（低峰期执行）
```

**优势**：
- **写操作极快**：只更新一个标记字段，不删缓存，无空窗期
- **无击穿风险**：缓存始终存在（只是标记失效），读请求不会同时穿透DB
- **容错强**：即使标记更新失败，下次读请求也会从DB读到新数据并重建缓存

**常见坑点**：
- **缓存膨胀**：失效标记的旧缓存会占空间，需要异步清理任务定期回收
- **标记更新的原子性**：必须保证"更新DB + 更新缓存标记"的一致性。建议：先更新DB，再更新缓存标记；失败时容忍短暂不一致
- **清理时机**：批量物理删除应在低峰期执行，避免与正常读写竞争

**失败恢复策略**：
- 标记更新失败 → 读请求仍从DB读取最新数据并重建缓存，自修复
- 异步清理失败 → 缓存空间膨胀但不影响功能，重试即可

**适用场景**：高频写、实时数据（订单状态流转、车辆实时位置、统计数据更新）

### 2.1 Binlog方案：Canal + MQ消费者配置

**Canal配置示例**（监听指定表的Binlog变更）：

```properties
# canal.properties
canal.serverMode = kafka           # 投递到Kafka
canal.mq.servers = kafka-cluster:9092
canal.mq.topic = cache-sync-topic

# 实例配置 - instance.properties
canal.instance.master.address = mysql-master:3306
canal.instance.filter.regex = product_db\\..*  # 只监听product_db的所有表
```

**MQ消费者核心逻辑**（伪代码）：

```java
@KafkaListener(topics = "cache-sync-topic")
public void onBinlogEvent(BinlogEvent event) {
    String cacheKey = buildCacheKey(event.getTable(), event.getPrimaryKey());

    switch (event.getType()) {
        case INSERT:
        case UPDATE:
            // 方案A：删除缓存，下次读时重建
            redis.del(cacheKey);
            // 方案B：直接更新缓存（更强一致，但需处理部分字段更新）
            // redis.set(cacheKey, event.getAfterData());
            break;
        case DELETE:
            redis.del(cacheKey);
            break;
    }

    // 失败重试由MQ机制保障（重试3次 → 死信队列）
}
```

**保证同Key有序消费**：

```java
// Kafka分区策略：按Key哈希分区，保证同一Key的事件有序
ProducerRecord<String, BinlogEvent> record =
    new ProducerRecord<>("cache-sync-topic", cacheKey, event);
// cacheKey作为分区Key，同Key必然落入同一分区，消费有序
```

### 2.2 版本号方案：缓存校验伪代码

**写操作**：

```java
// 更新DB，version自增
productDao.updateWithVersion(id, newData, currentVersion);
// SQL: UPDATE product SET data=?, version=version+1 WHERE id=? AND version=?

// 删除缓存（只删一次，无需延迟）
redis.del("product:" + id);
```

**读操作**：

```java
public Product getProduct(Long id) {
    String cacheKey = "product:" + id;

    // 1. 查缓存
    CacheValue cached = redis.get(cacheKey);
    if (cached != null && cached.getVersion() >= expectedVersion(id)) {
        return cached.getData();  // 缓存有效
    }

    // 2. 缓存失效或版本低 → 查DB
    Product product = productDao.getById(id);

    // 3. 版本校验后写回缓存
    if (product.getVersion() > (cached != null ? cached.getVersion() : 0)) {
        redis.set(cacheKey, new CacheValue(product, product.getVersion()));
    }
    // else: DB版本更低（从库延迟），丢弃旧数据，不写缓存

    return product;
}
```

> **版本号来源策略**：缓存中存储version，读请求用DB读到的version与缓存version比较。更严格的方案是在主库更新后，额外写入一个 `latest_version:{id}` 到Redis，读请求以此作为"最新版本基准"。

### 2.3 租约方案：Redis Lua脚本实现

**租约生成+校验 Lua脚本**（原子操作）：

```lua
-- get_with_lease.lua: 读缓存，Miss时生成租约
local key = KEYS[1]
local lease_key = KEYS[2]      -- lease:{key}
local lease_ttl = ARGV[1]      -- 租约有效期(ms)

local value = redis.call('GET', key)
if value then
    return {value, ''}          -- 缓存命中，无需租约
else
    local lease_token = redis.call('INCR', lease_key)
    redis.call('PEXPIRE', lease_key, lease_ttl)  -- 设置租约过期
    return {'', tostring(lease_token)}            -- Miss + 租约Token
end
```

```lua
-- set_with_lease.lua: 带租约校验写缓存
local key = KEYS[1]
local lease_key = KEYS[2]
local token = ARGV[1]
local value = ARGV[2]
local ttl = ARGV[3]             -- 缓存有效期(ms)

local current_token = redis.call('GET', lease_key)
if current_token == token then
    redis.call('SET', key, value)
    redis.call('PEXPIRE', key, ttl)
    redis.call('DEL', lease_key)  -- 租约消耗，防止重复写入
    return 1                      -- 写入成功
else
    return 0                      -- 租约不匹配，写入被拒
end
```

**写操作删除租约**：

```lua
-- invalidate.lua: 删除缓存+租约
local key = KEYS[1]
local lease_key = KEYS[2]
redis.call('DEL', key)
redis.call('DEL', lease_key)     -- 删除租约，阻断所有持有旧租约的写入
return 1
```

### 2.4 逻辑失效方案：标记+异步清理伪代码

**写操作（标记失效）**：

```java
// 更新DB
orderDao.updateOrderStatus(orderId, newStatus);

// 标记缓存失效（不删除，只改标记）
String cacheKey = "order:" + orderId;
redis.set(cacheKey, JSON.stringify({
    "is_deleted": true,
    "data": oldData   // 旧数据保留，但标记为失效
}));
```

**读操作（发现失效→回源重建）**：

```java
public Order getOrder(Long orderId) {
    String cacheKey = "order:" + orderId;
    String cached = redis.get(cacheKey);

    if (cached != null) {
        CacheValue cv = JSON.parse(cached);
        if (!cv.is_deleted()) {
            return cv.getData();       // 缓存有效
        }
        // 标记失效 → 回源DB
    }

    // 查DB获取最新数据
    Order order = orderDao.getById(orderId);
    // 重建缓存（标记为有效）
    redis.set(cacheKey, JSON.stringify({
        "is_deleted": false,
        "data": order
    }));
    return order;
}
```

**异步批量清理**：

```java
// 低峰期定时任务（如每小时执行）
@Scheduled(cron = "0 0 3 * * ?")  // 每天凌晨3点
public void cleanupStaleCache() {
    // 扫描标记is_deleted的Key（需配合Redis SCAN或维护失效Key列表）
    Set<String> staleKeys = redis.scanByPattern("order:*", "is_deleted=true");
    for (String key : staleKeys) {
        redis.del(key);
    }
}
```

## 选型篇：方案对比与决策指引

### 3.1 全维度对比表

| 方案 | 一致性 | 性能 | 业务侵入 | 实现复杂度 | 运维成本 | 适用场景 | 代表大厂 |
| ---- | ---- | ---- | ---- | ---- | ---- | ---- | ---- |
| 延迟双删 | 最终一致（不可靠） | 一般 | 高 | 低 | 高 | 中小流量、简单业务 | 早期中小厂 |
| Binlog异步 | 最终一致（可靠） | 极高 | 极低 | 高 | 中 | 读多写少、高并发 | 阿里、美团、字节 |
| 版本号校验 | 强一致 | 较高 | 中 | 中 | 低 | 订单、库存、账户 | 京东、拼多多 |
| 租约(Lease) | 强一致 | 高 | 中 | 中 | 低 | 热点Key、高并发读 | Meta、Facebook |
| 逻辑失效标记 | 最终一致 | 极高 | 中 | 中 | 中 | 高频写、实时数据 | 美团、滴滴 |

### 3.2 场景决策流程

```
你的业务场景是什么？
│
├─ 读多写少（如商品详情、用户信息）
│  └─ 允许短暂不一致？

│
├─ 写频率高（如订单状态、实时位置）
│  └─ 是否需要强一致？
│     ├─ YES → 【方案2: 版本号校验】
│     └─ NO  → 【方案4: 逻辑失效标记】✅ 最推荐

├─ 热点Key高并发读（如热搜、热门文章）
│  └─ → 【方案3: 租约(Lease)】✅ 专为防并发击穿设计

├─ 中��流量、简单业务、快速上线
│  └─ → 【延迟双删】（能用但不推荐，长期应迁移）
```

### 3.3 混合策略详解

单一方案往往不够，大厂实践多为**组合方案**：

#### 混合策略A：Binlog + 版本号校验（强一致 + 异步兜底）

```
核心：Binlog异步删除缓存作为主路径，版本号校验作为兜底防线

写流程：
  1. 更新DB（version+1）
  2. Binlog事件触发 → 异步删除缓存

读流程：
  1. 缓存命中 → 检查version → 版本低则回源DB
  2. 缓存Miss → 查DB → 版本校验后写回

兜底：即使Binlog消费延迟，读请求的版本校验也能阻止脏数据回填
适用：既要高吞吐又要强一致（商品详情页+库存查询）
```

#### 混合策略B：Binlog + 逻辑失效标记（高频写 + 异步兜底）

```
核心：高频写用逻辑失效标记避免击穿，Binlog异步兜底防止标记更新遗漏

写流程：
  1. 更新DB → 标记缓存失效
  2. Binlog事件 → 异步确认标记已失效（兜底）

读流程：
  1. 发现失效标记 → 回源DB → 重建缓存

兜底：写操作标记失败时，Binlog消费者再次标记/删除
适用：高频写且需要最终可靠一致（订单状态流转）
```

#### 混合策略C：租约 + 版本号（热点Key强一致双保险）

```
核心：租约防并发击穿，版本号防脏数据回填

读流程：
  1. 缓存Miss → 获得租约 → 查DB → 带租约+版本号写入缓存
  2. 租约校验 → 版本号校验 → 双重防线

写流程：
  1. 更新DB（version+1）→ 删除缓存+租约

适用：极高并发+必须强一致（秒杀库存、热门商品实时价格）
```

### 3.4 从延迟双删迁移到新方案

```
阶段1：在延迟双删基础上引入版本号校验（低成本改造）
  - DB表加version字段
  - 读请求增加版本校验逻辑
  - 延迟双删仍保留，但版本校验作为兜底防线
  - 效果：脏数据回填问题大幅减少

阶段2：引入Canal监听Binlog，替代第二次延迟删除
  - 部署Canal + MQ
  - Binlog事件触发缓存删除，替代固定延迟的第二次删除
  - 延迟双删的"第二次删除"不再需要
  - 效果：一致性大幅提升，延迟窗口消除

阶段3：完全移除延迟双删逻辑，Binlog+版本号独立运作
  - 业务代码移除延迟删除逻辑
  - 缓存维护完全由Binlog消费者+版本号校验承担
  - 效果：0业务侵入、高可靠、可追溯
```

> **迁移原则**：渐进式，不一步到位。每个阶段都比上一个阶段更可靠，且可以随时回退到上一个阶段。

## 实战篇：3个典型场景完整方案

### 4.1 场景一：商品详情页（读多写少，百万级QPS）

**业务特征**：
- 读频率极高（百万QPS），写频率低（商品信息更新少）
- 允许短暂不一致（用户看到旧价格几秒，业务可接受）
- 缓存是核心防线，击穿会压垮DB

**推荐方案**：Binlog异步事件驱动

**完整架构**：

```
用户请求 → Nginx → 应用服务 → Redis缓存
                              ↓ 缓存Miss
                            MySQL从库

商品运营 → 更新主库 → Binlog → Canal → Kafka → 缓存消费者 → 删除Redis缓存
```

**关键配置**：
- Canal监听 `product_db.product_info` 表
- Kafka Topic: `product-cache-sync`，分区数=Canal并行度
- 缓存消费者：按Key哈希分区消费，保证同Key有序
- 缓存TTL: 24小时（兜底过期，即使Binlog消费失败，缓存也会自然过期）
- 缓存消费者重试: 3次重试 → 死信队列 → 定时任务补偿

**容灾设计**：
- Canal宕机 → Canal集群HA切换，位点自动恢复
- Kafka消费延迟 → 读请求穿透DB，DB从库承载（读多写少，从库可扛）
- 极端情况：Canal+Kafka同时故障 → 缓存TTL兜底，24小时后缓存自然过期重建

### 4.2 场景二：订单/库存（强一致，写频率适中）

**业务特征**：
- 读频率较高（查订单状态、查库存），写频率适中（下单、状态变更）
- **不允许不一致**：库存超卖、订单状态错误直接影响营收和用户体验
- 主从延迟必须被考虑（库存查询可能查从库）

**推荐方案**：版本号校验 + Binlog异步兜底（混合策略A）

**完整架构**：

```
下单/变更 → 更新主库(version+1) → 删除缓存
                                    ↓ Binlog → Canal → MQ → 异步确认删除(兜底)

查订单/库存 → Redis缓存 → 版本校验 → 命中且版本有效 → 返回
                            ↓ 版本低或Miss → 查主库(强一致场景查主库!) → 版本校验写回
```

**关键决策**：
- **库存查询必须查主库**：库存数据不允许读到旧值，读请求强制路由主库
- **订单状态可查从库+版本校验**：从库延迟读到的旧数据因版本低被拒绝，不写缓存
- 缓存TTL: 30分钟（订单数据变化后需要及时更新）
- 版本号字段：`version INT NOT NULL DEFAULT 0`，每次更新 `version=version+1`

**容灾设计**：
- 缓存删除失败 → 版本校验兜底，旧数据被拒绝
- Binlog消费延迟 → 版本校验兜底，读请求不会回填脏数据
- 主库查询超时 → 降级为读从库+版本校验，短暂不一致但不会超卖

### 4.3 场景三：热点Key高并发读（热搜榜、热门文章）

**业务特征**：
- 极高并发读（单Key QPS可达10万+），写频率低（排行榜更新）
- **缓存击穿是最大风险**：热点Key失效瞬间，十万级请求同时穿透DB
- 一致性要求中等（排行榜允许短暂延迟）

**推荐方案**：租约(Lease) 机制

**完整架构**：

```
读请求 → Redis → 命中 → 返回
                ↓ Miss → 获得Lease Token → 查DB → 带Token写缓存
                ↓ Miss → 获得Lease Token → 查DB → Token已失效 → 写入被拒 → 返回DB数据(不缓存)

排行榜更新 → 更新DB → 删除缓存 + 删除租约Token
```

**关键配置**：
- 租约有效期: 50ms（根据业务P99 RT调整，保证正常请求能完成回填）
- 租约Key格式: `lease:{original_key}`
- 缓存TTL: 5分钟（排行榜数据时效性要求不高）
- Lua脚本执行：租约生成+校验+缓存写入全部原子操作

**防击穿策略**：
- 只有持有有效租约的请求能写缓存 → 并发读请求不会同时回填，避免回填风暴
- 被拒的请求直接返回DB数据 → 不缓存，不影响功能
- 热点Key永不过期策略：不设TTL，完全依赖写操作主动删除+租约机制更新

**容灾设计**：
- 租约Token丢失 → 自然过期（50ms），不会残留
- Lua脚本执行失败 → 降级为普通读写+短暂不一致窗口
- Redis宕机 → 全量穿透DB，需要DB有足够承载能力（热点Key场景需提前评估）

## 总结：从"时间妥协"到"事件驱动+确定性校验"

大厂弃用延迟双删，本质是架构升级的三重转变：

1. 从**被动等时间** → **主动感知数据变更（Binlog/事件）**，精准触发缓存更新，不赌延迟窗口
2. 从**暴力删缓存** → **版本/租约校验**，从根源防止旧数据污染缓存，兼顾一致性与性能
3. 从**业务耦合** → **异步解耦**，缓存维护下沉到中间件，业务专注核心逻辑，稳定性、可运维性大幅提升

**核心方法论**：不要用不确定性对抗不确定性——用确定性机制（事件驱动、版本校验、原子操作）替代不确定的时间窗口，这才是大规模分布式系统的一致性正道。
